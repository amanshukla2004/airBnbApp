# Nox Platform: CORS Resolution Documentation

## 1. The Problem: "Fake" CORS Errors
A common frustration in Spring Boot development is recurring CORS errors even after configuring `@CrossOrigin` or `WebSecurityConfig`. These are often not CORS issues at all, but rather "unhandled responses" that lack headers.

### Root Causes we addressed:
- **Redundant Configurations**: Having `@CrossOrigin` on controllers AND a global configuration in `WebSecurityConfig` causes header duplication or conflicts that browsers reject.
- **Security Filter Interruption**: If Spring Security blocks a request (e.g., 401 Unauthorized or 403 Forbidden) BEFORE the CORS logic runs, the response is sent without CORS headers. The browser then masks the 401/403 error and displays a "CORS Error."
- **Preflight (OPTIONS) Failures**: Browsers send an OPTIONS request before `PATCH`, `DELETE`, or custom-header `POST` requests. If the backend doesn't handle OPTIONS globally, it often fails.

---

## 2. The Solution: Multi-Layered Robustness

### A. Dedicated High-Precedence Global Filter
We moved the CORS logic out of the Security configuration and into a dedicated `GeneralWebConfig.java` class.
- **Ordered.HIGHEST_PRECEDENCE**: This ensures the `CorsFilter` runs at the very beginning of the Servlet filter chain, even before the `SpringSecurityFilterChain`. 
- **Origin Patterns**: We used `setAllowedOriginPatterns` (e.g., `http://localhost:[*]`) which is more robust than fixed lists for development.

### B. Consolidated Configuration (Single Source of Truth)
- We removed all `@CrossOrigin` annotations from controllers.
- We removed the `http.cors()` inline configuration from `WebSecurityConfig`.
- This ensures that only the `GeneralWebConfig` filter is responsible for CORS, eliminating conflicts.

### C. Authentication Entry Point & Access Denied Handler
Crucially, we updated `WebSecurityConfig.java` to handle security failures correctly:
- **AuthenticationEntryPoint**: Handles 401 (Unauthenticated) errors.
- **AccessDeniedHandler**: Handles 403 (Unauthorized) errors.
By routing both through our `GlobalExceptionResolver`, we ensure that even when a user is logged out or lacks permissions, the resulting error response is still wrapped with the correct CORS headers, allowing the frontend to see the actual error message.

### D. credential Support
- `setAllowCredentials(true)` is strictly maintained to support the `HttpOnly` refresh token cookie strategy.
- We used `*` for Allowed Headers and specific methods (`GET`, `POST`, `PUT`, `PATCH`, `DELETE`, `OPTIONS`) to ensure modern frontend libraries (like Axios) never encounter header-mismatch errors.

---

## 3. Summary of Files Affected:
1. `GeneralWebConfig.java`: The new "engine" for CORS.
2. `WebSecurityConfig.java`: Cleaned up to delegate CORS to the global filter.
3. `AirBnbAppApplication.java`: Cleaned of experimental CORS beans.
4. `HotelController.java` & `ManagerProfileController.java`: Cleaned of redundant annotations.
